## Vision

Make Claude Code a real, opinionated Delphi development partner вҖ” one that:

1. **Knows the language and the ecosystem.** Object Pascal idioms, VCL/FMX, FireDAC, Spring4D, RAD Studio versions, RTL nuances.
2. **Validates its own output.** Code generated by the plugin should compile and pass tests automatically вҖ” not just look right.
3. **Speaks the user's language.** Bilingual (pt-BR / en-US) by default, with parity in templates, labels and notifications.
4. **Stays out of the way until it's useful.** Auto-activation only on Delphi files; never on unrelated codebases.

## Phase 1 вҖ” Encoding UTF-8/ANSI (v1.6.0)

### Problem
Claude Code writes `.pas` / `.dfm` / `.dpr` / `.inc` / `.fmx` files as **UTF-8 without BOM**. Pre-XE2 Delphi versions treat these files as ANSI/CP1252 and produce mojibake (`ГғВЎ`, `ГғВ§`). Modern RAD Studio uses heuristic detection, which fails on files with few accented characters.

### Solution
A `PostToolUse` hook that injects a UTF-8 BOM (`EF BB BF`) into Delphi files immediately after Claude writes or edits them. An optional legacy mode converts to CP1252 for pre-XE2 projects.

### Deliverables
- Cross-platform hooks: `hooks/fix-delphi-encoding.ps1` (Windows) and `hooks/fix-delphi-encoding.sh` (Linux/macOS)
- New skill `delphi-encoding` explaining BOM, `{$CODEPAGE}`, and encoding behavior in `LoadFromFile`/`SaveToFile`
- `.delphi-dev.json` configuration file with `encoding_mode: "utf8-bom" | "cp1252"`
- Hook registration in `plugin.json`

### Open questions
- Default mode: `utf8-bom` (recommended for XE2+) or auto-detect from existing files?

## Phase 2 вҖ” Mobile FMX in `/new-project` + `delphi-fmx` skill (v1.7.0)

### Part A вҖ” `delphi-fmx` skill
Today, the FireMonkey knowledge base (color notation, mobile dimensions, deployment, FireDAC drivers, IMEI/device ID, SSL/TLS on Android, etc.) lives only locally on the maintainer's machine. Phase 2A ports that knowledge into the plugin so every user gets it.

### Part B вҖ” Knowledge base sync
A `scripts/sync-fmx-kb.ps1` script copies the canonical FMX knowledge files into `skills/delphi-fmx/references/` before each release. The local knowledge base remains the source of truth; the plugin ships a snapshot.

### Part C вҖ” Mobile support in `/new-project`
Extend `/new-project` to ask:
- Target platforms (Windows / Android / iOS / macOS / Linux)
- For mobile: orientation, local database (SQLite + FireDAC), permissions, push notifications, splash and icons
- Generate forms with correct `ClientWidth = 400` / `ClientHeight = 750`
- Unit names without dots (Android `ld.lld` requirement)
- DataModule with explicit FireDAC driver registration + `ConsoleUI.Wait`
- Pre-configured deployment for assets (Android `.\assets\internal\`, iOS `StartUp\Documents\`)

### Deliverables
- `skills/delphi-fmx/SKILL.md` + 13 reference files (bilingual: 26 files)
- `scripts/sync-fmx-kb.ps1` with `--dry-run`
- Extended `/new-project` flow for FMX mobile

## Phase 3 вҖ” Spring4D (v1.8.0)

### Goal
Teach Claude to recognize and generate code that uses **Spring4D** for Inversion of Control / Dependency Injection вҖ” registration, lifetimes, constructor injection, lazy resolution, named registrations.

### Deliverables
- `skills/delphi-spring4d/SKILL.md` + reference docs covering the Register вҶ’ Build вҶ’ Resolve cycle, lifetimes (Singleton, Transient, SingletonPerThread), constructor vs `[Inject]` injection, `Lazy<T>`, named registrations, and anti-patterns (e.g. avoiding `GlobalContainer`)
- Updates to `delphi-writer` (services and repositories suggest `Bootstrap.pas` registration)
- New audit dimension in `delphi-auditor`: **Inversion of Control**
- Updated `delphi-tester` with container-based mocking patterns

## Phase 4 вҖ” Build & Validation (v1.9.0, candidate for earlier)

### Problem
The plugin generates Delphi code today, but cannot **validate that the code compiles**. Capturing `msbuild` / `dcc32` `stdout` directly through the Bash or PowerShell shell is fragile on Windows: mixed encoding, truncated buffers, `cmd.exe /c` not inheriting the working directory.

### Solution
Three coordinated components:

#### A) `delphi-build` skill
Auto-activates on `.dproj` / `.dpr` / `.dpk`. Knowledge base covers RAD Studio versions (Studio 22.0 / 23.0 / 37.0 вҶ’ Delphi 11 / 12 / 13), the `rsvars.bat` mechanism, `msbuild` vs `dcc32` trade-offs, the `.bat` template that redirects output to a log file, and a catalog of compiler errors mapped to missing `uses` units.

#### B) `delphi-builder` agent
1. Detect installed Studios (scan `C:\Program Files (x86)\Embarcadero\Studio\*\bin\rsvars.bat`)
2. Choose the right one (or honor `studio` in `.delphi-dev.json`)
3. Generate `build.bat` if missing (template from the skill)
4. Execute via `cmd.exe /c '"<abs>\build.bat" 2>&1'`
5. Read `build_log.txt` with the `Read` tool (do not trust `stdout`)
6. Parse compilation errors (`E2003`, `E2065`, `F2613`)
7. Suggest a fix from the error catalog or by reading the offending file
8. Optional `--auto-fix` loop: apply, recompile, repeat until success or limit

#### C) `/build` command
```
/build                    # Debug / Win32, current repo
/build --release          # Config=Release
/build --plat=Android64   # cross-compile mobile
/build --auto-fix         # delegate to the agent with auto-correction
/build --tests            # compile + run DUnitX suite
```

### Integration with existing features
- `/write` вҖ” optional pre-validation before delivering code to the user
- `/tdd` вҖ” actually compile and run the generated tests
- `/audit` вҖ” new "Compilability" dimension
- `delphi-claudeignore` вҖ” auto-add `build_log.txt` / `test_log.txt` to the deny-list

### Open questions
- Detect Studio automatically and ask if multiple versions found, or require `.delphi-dev.json` upfront? (Default proposal: auto-detect, ask only on conflict)
- Ship `--auto-fix` in the first release or defer to a follow-up?

## Out of scope

These items have been considered and explicitly **declined** to keep the plugin focused:

- Support for Pascal dialects other than Object Pascal (Free Pascal/Lazarus is **not** a target вҖ” the conventions differ on key points)
- IDE integration beyond what Claude Code's plugin system provides (no RAD Studio plugin)
- Automatic refactoring of legacy projects (the `/audit` command produces a **report**, not autonomous mutations)
- Code generation that bypasses the Delphi style guide (the plugin's value is the discipline; an "off mode" would defeat that)
